# COGS Stack: Elevator Pitches

## Cowrie

Cowrie is a binary serialization format for structured data, available in two variants: Gen1, a lightweight tag-length-value codec with 11 core types and 6 graph types, and Gen2, a full codec with 18+ types including native tensors, images, audio, graph shards, dictionary coding, and optional gzip/zstd compression. Gen2's dictionary coding reduces payload size by roughly 50% for data with repeated schemas (e.g., 1000-object arrays go from 48KB JSON to 23KB Cowrie), while Gen1 provides single-pass encoding with predictable latency for real-time use. Implementations exist in Go, Rust, Python, C, and TypeScript, with cross-language binary compatibility verified through shared test fixtures. Cowrie is for anyone building ML pipelines, GNN systems, or high-throughput APIs who needs a binary format that handles tensors, graphs, and ordinary structured data in a single codec.

## Glyph

Glyph is a text serialization format that encodes the same data as JSON using 40-60% fewer LLM tokens while remaining human-readable. It eliminates quotes, colons, and commas (`{name=Alice age=30}` instead of `{"name":"Alice","age":30}`), and its auto-tabular mode compresses homogeneous lists into column headers plus rows, achieving 50-62% token savings on batch data like search results and tool outputs. Glyph supports streaming validation, allowing malformed tool calls to be detected and cancelled mid-generation rather than after the full response, and includes state fingerprinting via SHA-256 for concurrent modification safety. Implementations are available in Go, Python, JavaScript, Rust, and C, with bidirectional JSON conversion. Glyph is for developers building LLM-based agents and tool-calling systems where token cost, context window utilization, and streaming error detection are practical concerns.

## Shard

Shard is a binary container format for storing collections of named entries with O(1) hash-based lookup, per-entry CRC32C checksums, and 64-byte data alignment for SIMD and GPUDirect Storage compatibility. The format uses a fixed 64-byte header and 48-byte index entries (xxHash64-keyed), supports per-entry zstd/lz4 compression and AES-256-GCM encryption, and enables streaming writes without seek-back. Shard defines role profiles for different workloads: MoSH for model weights (with native LoRA delta entries and mixed quantization per tensor), SampleShard for training data, GemmPanel for distributed GEMM tiles, and W-SHARD for world-model episodes. Implementations exist in Go, Rust, Python, C, and TypeScript, all binary-compatible. Shard is for ML engineers and infrastructure teams who need a single container format that handles model distribution, training checkpoints, and dataset storage with integrity verification and random access that existing formats do not provide together.

## COGS (Cowrie + Glyph + Shard)

COGS is a three-layer open-source data format stack where each layer handles a distinct concern: Cowrie serializes individual values (tensors, graphs, structured data) into compact binary, Glyph serializes the same values into token-efficient text for LLM consumption, and Shard packages collections of Cowrie-encoded or raw entries into indexed, checksummed container files. The layers compose naturally: a Shard file stores Cowrie-encoded tensors as named entries, Glyph encodes metadata and tool-call payloads at the text layer, and all three share the same type system across five language implementations (Go, Rust, Python, C, TypeScript). This means a single stack covers the path from model weights on disk (Shard) through binary wire protocol (Cowrie) to LLM agent communication (Glyph), without format translation at the boundaries. COGS is for teams building end-to-end ML systems -- training pipelines, inference servers, agent frameworks -- who want one consistent set of formats rather than stitching together Protocol Buffers, JSON, SafeTensors, and GGUF.
